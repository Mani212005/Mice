Where this actually stands today

All three symptoms you named trace to specific, findable code — not to vague "the model is dumb." Verified against the working tree, not assumed.

Bottleneck 1

A confirmation on every click

autopilot_axi() prints "Do this one action? [y/N]" before every action, capped at 6 actions per run — regardless of whether the action is a harmless scroll or a real click. It doesn't consult the ToolKind::ReadOnly / Mutating split that already exists one file over.

Bottleneck 3

What's already good and shouldn't be rebuilt: chrome-devtools-axi's snapshots are already close to the accessibility-tree observation format the research below converges on (uid-based, not raw DOM), and already surface a clean deterministic failure signal (STALE_REF, via generation-prefixed uids like g3:). There's also already a local→cloud escalation cascade (ExecutionLane::Local → CheapCloud → Frontier, AXI_LOCAL_UNCERTAINTY_LIMIT = 2 uncertain turns before falling back). None of that gets thrown away — the proposal below builds directly on it.

The core reframe: Record → Replay → Repair

This is the single highest-leverage idea in the research, and it predates LLMs entirely. IBM's CoScripter (2007) and its spreadsheet successor Vegemite recorded a browser demonstration once and replayed it across many rows of input — structurally identical to "find 1,000 restaurants." Modern commercial RPA (UiPath's "Healing Agent") does the same thing with an LLM added at exactly one point: only when a selector breaks, to repair it — never to drive routine steps. Applied to MICE, gemma3:4b stops being the thing that decides step 2 through step 1,000. It becomes the thing that decides step 1, and the thing that gets called back in for the rare step where something changed.

The model is in the loop for the plan, the first row, and any repair. For rows 2 through 1,000 it is, by design, mostly out of the loop — which directly targets the failure mode you're hitting: a 4B model doesn't get a chance to hallucinate or lose track of state on a page it isn't being asked to freshly reason about.

Four supporting pillars

Each maps to a specific, findable line in this repo — none of these require a bigger model.

Pillar A

Tiered, batched confirmation

Fixes the "[y/N] on every click" complaint directly. Wire the existing ToolKind split into the axi confirm gate: read-only/navigation auto-runs and logs; mutating actions batch into a checkpoint ("about to do these 4 things — proceed?") instead of confirming per-action. Do not gate on the model's own self-reported confidence — the research is explicit that models learn to report high confidence below their actual competence ("Behavioral Credibility Trilemma"). Gate on the action's reversibility, a property of the tool, not the model's opinion of itself. Risk-tiered autonomy of this shape measured a 66.9% incident reduction vs. unrestricted autonomy in the AURA framework — and your own plan v5 already specified almost exactly this UX ("consent once per goal… Esc stops instantly… per-action confirmation survives as an opt-in careful mode") before the axi engine hardened it back into confirm-everything. This pillar is largely finishing what plan v5 already decided, not inventing new UX.

Pillar B

Make hallucinated actions structurally impossible

Two changes, both cheap. First: grammar-constrained decoding (GBNF, native to llama.cpp / usable through Ollama) forces the sampler to only emit tokens on a valid path through the ToolDecision JSON schema — the model cannot emit a malformed action, which is a harder guarantee than the current prompt-text instruction ("clarify fill action JSON instruction for small models") can ever give. Second: a free, deterministic pre-flight check — does this uid still exist in the current snapshot — before firing the action, not after. This is the same idea as STALE_REF, just proactive instead of reactive, and it's the direct fix for the "tolerate dynamic UID shifts" and "looping" bugs already in your commit history.

Pillar C

A local recipe library that compounds

Voyager (Minecraft agents, 2023) keeps a vector-indexed skill library so an agent retrieves and adapts a previously-proven action program instead of re-deriving it cold every time. Applied here: every time MICE figures out "how to search Google," "how to open a Sheet and append a row," it's saved locally, keyed by an embedding of (site + goal shape). The next time any goal needs that step, gemma3:4b's job shrinks from "invent this" to "does this recipe still apply" — a classification-shaped task, which is close to what small models are actually good at. This is also the feature that makes MICE feel like it's learning the more you use it, with zero fine-tuning and zero data leaving the laptop.

Pillar D

CAPTCHA posture: reduce, then hand off

No evasion — the honest finding from the research is that the only reliable answer to an actual challenge is a clean human handoff at that exact moment, which AgentAction::Handoff already models. What's worth adding: default autopilot to the existing CHROME_DEVTOOLS_AXI_AUTO_CONNECT attach-to-real-Chrome mode (a logged-in, cookie-bearing, non-headless profile is the biggest single anti-bot-trigger reducer) and detect CAPTCHA-shaped pages specifically to trigger handoff rather than burning retries against them.

Suggested build order

Numbered to continue the existing plan-doc series (M13 tool registry and M14 axi engine are already built). Ordered by relief-per-line-of-code — A is the smallest change with the most immediate effect on your stated complaint.

M17a

Wire ToolKind into the confirm gate

Smallest change, biggest immediate relief — the two pieces already exist, they're just not connected. Add checkpoint batching.

M17b

Grammar-constrained decisions + pre-flight verify

Replace the prompt-text JSON instruction with a real GBNF grammar; add the uid-exists check before execution, not after.

M17c

Recipe / skill library

On-disk, embedding-keyed store of successful step sequences; record on every success, retrieve-and-adapt before asking the model to plan cold.

M17d

CAPTCHA-aware handoff

Default to attach-to-real-Chrome for autopilot; detect challenge pages and hand off cleanly instead of retrying.

Risks and open questions

Recipes go stale when a site redesigns

Expected, not a flaw — this is exactly what the STALE_REF-triggered repair path is for. A stale recipe fails loudly and gets one bounded repair call, not a silent wrong action. Worth deciding: does a repaired step overwrite the old recipe automatically, or queue for a quick human glance first?

This doesn't make gemma3:4b smarter at novel sites

Honest framing: the win here is architectural — fewer chances for the model to be wrong, not a higher ceiling on how well it reasons about a page it's never seen. First-time-on-a-new-site tasks still hit the model's real limits. This is why the plan pass and teach-on-row-1 still matter, and why escalation to the existing CheapCloud/Frontier lane on repeated uncertainty stays in place as the safety valve.

Recipe retrieval needs local embeddings

Still small and feasible (embedding models in the tens-of-millions-of-parameters range run fine alongside gemma3:4b), but it's a new local dependency worth sizing before M17c starts, not assumed for free.

Ollama grammar support needs local verification

GBNF is native to llama.cpp; Ollama's own API surface for constrained JSON output should be re-checked against the currently installed Ollama version before M17b is scoped as "just wire it up."
