MICE product plan · Mission Control · 22 July 2026
Decision: evolve coordination into a plan-driven mission controller

Give MICE a plan; it turns compatible agents into a managed, visible team.

A developer points MICE at a plan in plan/, chooses installed harnesses such as Codex, Claude Code, and a configured Antigravity adapter, reviews a proposed task graph once, then launches isolated worktrees and terminals. MICE observes Git, tracks genuine completion, predicts merge risk early, and offers the next safe task.

RecommendationBuild Mission Control as a daemon-owned orchestrator with a Ratatui client—not as a generic shell script launcher. The initial promise is “one reviewed launch,” not unchecked autonomous delegation.
What changes

Existing pieces are useful, but they are not yet a mission system.

ReuseMCP bridge

mice mcp-server already connects Codex and Claude-compatible harnesses to MICE tools.

ReuseGit observer

mice team risks already finds registered worktrees and hunk-level overlap without changing Git.

ReuseTerminal stack

The existing Ratatui/Crossterm review interface provides the terminal UI base.

BuildMission owner

Task graph, adapters, worktree creation, lifecycle truth, notifications, and safe reassignment do not exist yet.

Critical boundary: mission state must be scoped by the Git common-directory hash and mission ID. The current shared memory is not sufficient by itself because it is not repository/mission isolated.
Target architecture

The MICE daemon owns the mission; terminals are controlled participants.

The daemon owns lifecycle and native notifications through existing Rust↔Swift IPC. The TUI is a client, so closing a terminal does not abandon a mission. Agents retain private transcripts; MICE stores only bounded, inspectable coordination facts and Git-derived evidence.

Git wins on facts

Agent reports provide intent. Git worktree identity, merge base, changed paths, rename aliases, hunks, and checks determine visible truth.

Adapters, not assumptions

Codex and Claude Code get first-party adapters. “Antigravity” remains a detected/configured adapter until its CLI, MCP support, and follow-up behavior pass a probe.

Bounded context

Each agent gets its task card, dependencies, relevant decisions, and current warnings—not another agent’s transcript or a duplicated plan.

One reviewed launch

The happy path is fast without silently taking control.

  1. Choose a plan. Run mice mission plan plan/release.md --agents codex,claude,antigravity from a Git worktree.
  2. Preflight. Verify clean base, Git common directory, available worktree paths, harness capabilities, and MCP connectivity.
  3. Propose segregation. A model emits strict JSON: objective, acceptance checks, dependencies, predicted paths, and preferred capability. Deterministic validation rejects cycles, missing acceptance, or unsafe parallel scope.
  4. Review once. The TUI shows mapping, branches, dependency order, and predicted risks. Enter launches approved ready tasks; --launch is the explicit noninteractive equivalent.
  5. Monitor and hand off. MICE opens terminals, observes actual Git changes, notifies on verified completion, and suggests one safe next task for an idle agent.

Lifecycle contract

“Process exited” is not “task finished.”

New MCP mission tools

mission_context, task_heartbeat, task_report, task_blocked, and mission_notice_ack; each validates its task and mission token.

Completion evidence

VerifiedReady requires an agent report plus configured checks and inspectable Git state. An unreported exit never produces a success notification.

Push honestly

Adapters may deliver notices through documented session controls. Other harnesses see notices on their next MCP request and in the TUI—MICE never types text into arbitrary terminals.

Implementation sequence

Ship truth and recovery before convenience.

M0 · Contracts
Dry-run mission model. Typed IPC/MCP mission types, Git-common-dir identity, strict task graph validator, agent probe, and mice mission plan --dry-run. No worktrees or agent launch.
M1 · TUI
Reviewable control room. Reuse Ratatui to show the plan, graph, assignment, scope, dependencies, preflight, and errors. Persist only metadata outside the repository.
M2 · One harness
Codex end-to-end. Provision owned worktrees/branches, launch through a no-shell adapter and selectable terminal launcher, inherit mission MCP context, and recover after TUI restart.
M3 · Coordination
Live risk and notification. Bounded Git monitor, heartbeats, task reports, notices, Swift completion notification, and manual safe-next-task approval.
M4 · More harnesses
Claude Code, then Antigravity. Add an adapter only after conformance checks prove launch, worktree isolation, MCP access, lifecycle observation, and safe follow-up delivery.
M5 · Trust calibration
Optional automation. Only measured accuracy unlocks opt-in automatic launch of dependency-ready, declared-disjoint tasks. Automatic merge, rebase, and cross-worktree edits remain out of scope.
Acceptance gates

Prove it removes a human bottleneck instead of moving one.

Non-negotiable tests

  • Dirty base repository refuses launch and changes neither Git nor worktrees.
  • Each worktree has an owned mission branch and is recoverable or explicitly removable.
  • Different worktrees resolve one repo-scoped mission context; separate repos never share one.
  • Rename-vs-old-path and same-hunk changes become Red; malformed, oversized, or timed-out scans are Unassessed.
  • An agent crash becomes ExitedUnreported, never “finished.”
  • Credentials, capture/clipboard content, model weights, user config, and agent transcripts never enter the repo or mission ledger.

Decisions before M2

Three choices determine the implementation shape.

Terminal

Where do agents run?

Recommended: selectable tmux or macOS Terminal tab launcher, plus headless mode. Never build a shell command from plan text.

Policy

How automatic is reassignment?

Recommended: human-approved suggestions by default. Later opt-in automation only launches dependency-ready, declared-disjoint work.

Adapter

What is Antigravity?

Recommended: discover its installed CLI before coding. Until then it is a configurable external harness—not a promised built-in.

This plan preserves MICE’s native boundary: Rust owns planning, state, Git, routing, and process control; Swift owns macOS notifications/UI. Design matches the existing MICE coordination-plan artifact because the product has no web design system to inherit.
